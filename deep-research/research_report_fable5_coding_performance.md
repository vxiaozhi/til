# Claude Fable 5 实际编程表现深度分析

## 摘要

Claude Fable 5 在实际编程任务中的表现呈现显著的"任务分化"特征。短代码写作场景下与 Opus 4.8 差距不明显，开发者甚至觉得"aha moment 不明显"；但在迁移重构、遗留代码治理、长周期自主开发等任务中，展现出了与前代模型完全不在同一量级的能力差距。核心发现包括：Fable 5 是目前唯一能在复杂全栈项目中实现零修改交付的模型；在安全修复领域解出了 4 道此前无任何模型能解的漏洞问题；但其安全护栏导致的过度拒绝（ProgramBench 200 题全部拒答）和安全分类器误触发问题严重影响了日常可用性。用户实际测试费用为 Opus 4.8 的 3 倍、GPT-5.5 的 8 倍，但能以低 effort 档位超越 Opus 4.8 的 xhigh 档位表现。

## 一、任务分化：短代码 vs 长周期工程

Fable 5 在实际编程中最为显著的特征是能力随任务复杂度非线性增长。日常编码场景下，开发者普遍反馈差异不大。代码补全、函数生成、API 调用等常规任务中，Fable 5 与 Opus 4.8、GPT-5.5 的差距并不悬殊。开发者描述为"aha moment 不明显"，因为此类能力各模型均已日趋成熟，期望值已被拉至高位。

但在迁移、重构、遗留代码治理等长周期任务中，差距呈指数级扩大。一位开发者分享了典型体验：让 Fable 5 重构一个 5000 行的遗留 Python 模块，它不仅准确识别了所有依赖关系，还主动提出了三处潜在的性能优化点。这种全局理解加主动建议的能力，被描述为之前的模型完全不具备。一位资深架构师的评价颇具概括性："新代码是想象力的游戏，存量代码是侦探的游戏。Fable 5 更像是一个侦探。"

Anthropic 官方定位也印证了这一特征："这代模型不是用来聊天的，是用来干长活、干重活的。"其核心价值主张不是回答"如何写这段代码"，而是解决"如何让这个项目持续健康运行下去"。

数据层面，Arena.ai 的评测提供了量化佐证。Fable 5 在用户确认任务完成率指标上达到 18.2%，领先 Opus 4.8 和 GPT-5.5 前所未有的大幅差距；好评与投诉比达 30.6%（每 30.6 个好评对应 1 个投诉），表明其不仅完成任务而且完成得让用户满意。

## 二、全栈项目实测：鱼皮的两轮硬核对比

程序员鱼皮于 Fable 5 发布次日进行了目前最详细的公开对比测试，在同场竞技条件下对比了 Fable 5、Opus 4.8 和 GPT-5.5。

### 第一轮：TaskFlow 全栈项目一把梭

项目为任务管理看板，包含用户注册登录、三列看板拖拽、任务增删改查、数据图表面板、搜索筛选、暗色亮色主题切换和响应式设计共 7 个功能需求，技术栈为 React 加 TypeScript 前端，Python FastAPI 后端，SQLite 数据库。三个模型使用相同提示词，全部开到 High 思考档位，全程零人工干预。

三个模型最终都能把项目跑起来，但关键差异在于交付质量。Fable 5 是唯一做到零修改跑通的模型：TypeScript 编译一次通过、后端一次启动成功、全部 API 测试一次通过，做到真正的开箱即用。它还用 CDP（Chrome DevTools Protocol）合成了真实的鼠标拖拽事件，在浏览器里实测了看板拖拽的持久化效果，验证深度远超其他模型。

Opus 4.8 架构分层最规范、UI 设计感不错，但有几个小 Bug 需要修复才能跑起来。GPT-5.5 做得快但做得糙，界面偏简陋，细节处理不到位（看板列标题直接使用英文）。

鱼皮总结：Fable 5 的优势主要体现在工程可靠性上，这种"交付确定性"对效率的影响非常大，少一次来回调试可能就省了半小时。

### 第二轮：重构 Claude Code 源码

这是测试的核心重头戏。测试者将泄露的 Claude Code 约 50 万行源码提供给模型，要求自主分析架构设计，然后从零重构一个能在终端实际运行的命令行 AI 编程助手（命名为 Yupi Code），全程不人工干预。

这是最能暴露模型差异的测试。测试结果和费用如下：

| 指标 | Fable 5 | Opus 4.8 | GPT-5.5 |
|------|---------|----------|---------|
| 交付可用产品 | 是（唯一） | 否 | 否 |
| 总费用 | $38.66 | $13.38 | $4.61 |
| 总 Token | 2146 万 | 1686 万 | 531 万 |
| 是否自动适配本地配置 | 是（自动读取本地 DeepSeek 配置） | 否（需要 Anthropic Key） | 否（需要 Anthropic Key） |
| 交互测试 | PTY 真实终端交互调试 | Mock Server 非交互环境 | 无实质性测试 |
| 开箱即用 | 是 | 需二次修复 | 不能正常使用 |

三个模型的实际表现：

Opus 4.8 写了最多的测试，通过模拟 Mock Server 跑通了测试流程，自主验证层次最多。但所有验证都在非交互环境下进行，从未在真实终端里验证过交互效果。结果到用户手里，输出显示就出了问题——界面风格与原装 Claude Code 有明显区别，AI 输出内容不能正确显示。

GPT-5.5 完成任务速度最快，但生成后需要 Anthropic API Key 才能运行，且输出信息比 Claude 精简很多。复用本地 Claude Code 配置后虽然能对话，但界面极其简陋。更严重的是，基本的 Read 工具调用报错，无法读取本地文件。

Fable 5 直接读取到了本地 Claude 配置，复用已配置好的 DeepSeek 国产模型，不需要 Anthropic API Key。功能全部正常——能普通对话、Agent 模式运行和工具调用，一次交付就能用，不需要任何二次修复。

开发过程对比揭示了深层差异。Fable 5 是唯一做了 PTY 终端交互式测试的模型：它用 PTY 在真实终端里反复调试交互，验证了 help、cost、权限对话框、写文件的全流程。它花了大量轮次调通了 PTY 下回车键处理的问题（\r 到 \n 转换），修复了 API 协议 Bug。这些投入看似费时费钱，但最终换来了最好的用户体验。

鱼皮的结论点出了核心命题：AI 自己测试通过与用户实际能用之间，隔着一个巨大的鸿沟。在 CLI 这种场景下，在真实环境中调通交互，远比在隔离环境中跑测试更能保障最终质量。

五维度综合评分：Fable 5 8.3 分、Opus 4.8 7.2 分、GPT-5.5 5.4 分。Fable 5 不是每项都最强——Opus 4.8 在架构理解和工程质量上略胜一筹——但唯一能交付可用产品这一条就压过了其他模型。

## 三、中文社区更多实测案例

除了鱼皮的系统性对比，中文社区还有其他高价值的独立实测。

花生 AI 做了一次从 Opus 4.8 到 Fable 5 的迁移式测试。先让 Opus 4.8 搭建 Electron App 基础版（文件浏览、内嵌终端），核心体验跑不通。换 Fable 5 后进行大改——代码编辑器、Markdown 所见即所得、图片标注、整体布局重构。作者的总结值得全文引用："不是更快，是犯错更少，一次就把问题解决。"具体表现为图片缩略图点击卡顿问题，以前需要模型猜原因、改版、再猜、再改，三四轮下来代码越改越乱。Fable 5 一次性诊断出两个叠加根因——缩略图实际上加载了完整原图加上每次点击都在重建文件网格——一次改完，点击响应从卡顿数秒降到 0.1 秒内。在 xterm.js 宽字符处理问题上，Fable 5 直接命中偏门知识，指出需要 unicode11 addon 并提醒这是实验性 API 需显式开启。作者引述："类似准确命中偏门问题的瞬间，一天里出现了很多次。"花生 AI 还指出了一个重要观察：Fable 5 的慢和准很可能是同一枚硬币的两面——动手之前想得久，猜错率就低，猜错率低就不用返工。它省掉的不是打字时间，是返工。

腾讯新闻苍何实测了完整的研发场景。他将 GitHub 仓库代码丢给 Fable 5，让它在 Claude Code 中基于 WeSight 开发一个 Obsidian Codex 插件。"几乎 2 个多小时，就帮我完成了需求"，插件集成了底层能力、自动检测本地 Claude Code 和 Codex 和 OpenCode 环境。作者说"震撼是真的挺震撼的"，但随即吐槽了 token 消耗的极端程度："几个简单任务，token 就哗哗的用，就分析完 WeSight 项目改了几个 bug，我的 5 小时额度就已经用完了。"想让它找 WeSight 的漏洞时，Fable 会直接拒绝请求——暴露了安全护栏对正常安全研究任务也构成阻碍。关于中文写作能力，苍何对比评价 Fable 5"比之前的 4.7、4.8 要好不少，有 4.6 那味了"。

知乎量子位的首日实测汇集了社区中最具对比性的两个案例。正面案例：一位开发者用 Fable 5 清理项目冗余代码，"直接删掉 7000 行无效代码，精简完成后整套系统运行丝滑，原有功能全部完好保留"——但代价是耗掉了该开发者 30% 的使用额度。反面案例：另一位开发者把老旧代码库交给 Fable 5 做全面重构，模型一次性触发 67 次工具调用，生成超百万行全新代码，新增 24 个文件，完成了架构拆分和模块化改造。整理后的代码结构规整、观感极佳，但这套精心重构后的代码最终无法正常运行。这是目前公开记录中最直接的过度重构导致失败的案例。

V2EX 社区（约 10 条真实回复，7266 浏览量）的反馈反映了真实的日常使用感受。正面反馈包括：Kokomashiro 做前端开发时直接把 Figma 设计图交给 Fable 5，做到了1 比 1 像素级还原，称之为"爆杀 5.5"；shyrock2026 形容"感觉提升巨大，那种轻快的、指哪儿打哪儿的感觉好像飞起来了"；qiaobeier 用 Fable 5 review 代码时，模型会自动打开 Jira 评估原始需求，"之前没有这种操作"；xiangran0028 报告"解决了工程中遗留的一个复杂问题，之前 Claude 4.8 和 Codex 5.5 尝试了几轮都不行"。负面反馈来自 dongmian2019 的"没感觉和 4.8 有什么大的差别"，以及多人反映的 token 消耗过高问题，简单的 CRUD 类任务被认为不值得用这么高级的模型。

## 四、迁移重构：真正的杀手锏

多个来源共同指向同一个结论：Fable 5 的真正优势不在写新代码，而在迁移重构和遗留代码治理。

官方公布的标志性案例中，Stripe 使用 Fable 5 在一天内完成了 5000 万行 Ruby 代码库的全库迁移，同样工作由人工团队完成预计需要两个多月。这在旧模型中即使理论上可能也需要大量的人工干预和分步执行，Fable 5 首次实现了这种规模的自主端到端执行。

社区实测中也有类似案例。一位开发者分享：让 Fable 5 将从 AngularJS 到 React 的项目迁移持续了 6 个小时的交互，模型始终保持着上下文一致性，不仅完成了代码转换，还主动更新了测试用例和文档。这种在超长周期中维持上下文一致性的能力，是此前所有编程模型都无法企及的。

跨语言编程与系统级代码生成方面，Fable 5 的表现同样突破。它能一次性设计多模块 SaaS 系统、生成跨文件一致性代码、编写自动化 Agent 工作流。实测中，重构 10 个文件的前端组件库时，Fable 5 一次性给出全部改动方案且相互引用正确，而 Opus 4.8 需要分多次对话完成。对于 10 步以上的 Agent 自动化长链路任务，Fable 5 步骤完成率超过 90%，Opus 4.8 在超过 10 步后失误率明显上升。

## 五、安全修复编程：Endor Labs 评测

Endor Labs 的 Agent Security League 基准评测了 Fable 5 在 200 个真实漏洞修复任务上的表现，这直接关联到编程中最敏感的安全修复场景。

Fable 5 成绩为 FuncPass（功能正确性）59.8%、SecPass（安全性正确性）19.0%。这两个数字绝对值不高，但关键在对比语境：此前顶级模型在 SecPass 指标上的最好成绩仅为个位数，而 Fable 5 达到了 19.0%，意味着在近五分之一的漏洞修复任务中，既消除了安全威胁又保持了代码功能正常运行。

更重要的突破是，Fable 5 解出了 4 道此前无任何模型能解的题目，涉及复杂的漏洞模式，需要模型同时理解攻击原理、代码逻辑和修复策略。这表明 Fable 5 在安全推理能力上已经跨过了某个关键门槛。

然而，59.8% 的 FuncPass 和 19.0% 的 SecPass 也说明距离完全可靠还有相当距离。AI 辅助漏洞修复的实际边界仍然是：AI 提出建议，人类做最终确认。Fable 5 的作用是将这个边界向前推进了一大步，但尚未突破。

需要特别注意的是，安全护栏对进攻性安全能力造成了严重影响。在 ExploitBench 网络安全评测中，无护栏的 Mythos 5 达 78%，而 Fable 5 因分类器拦截实际有效能力仅为约 40%，退回到 Opus 4.8 水平。在 HLE 评测（带工具）中，Mythos 5 得分 64.5%，Fable 5 实际有效约 57.9%，同样接近 Opus 4.8。这意味着在受护栏保护的领域，用户支付 Fable 5 的价格但实际获得的仍是 Opus 4.8 级别的能力。

## 六、英文社区的高级用户实测

英文开发者社区在 Fable 5 短暂可用的 4 天内留下了极具价值的实测记录。

Simon Willison（Django 联合创始人）在 Claude Code 中用 Fable 5 进行了 5.5 小时的密集开发。他用 Fable 5 为 datasette-agent 和 llm 项目添加新功能，Fable 5 几乎零交互往返完成了 llm 0.32a3 的大部分代码编写。它还自主发现并修复了 asyncinject 依赖库中的一个 Bug，并在极少提示的情况下反向工程了 Cloudflare WAF 并创建了自定义安全规则。Willison 最大的抱怨是成本——一个 CSS 滚动条 Bug 修复花费了约 12 美元，因为 Fable 5 没有简单地修补一行代码，而是自主搭建了一整套跨浏览器测试环境来验证修复效果。他的总结是 Fable 5" relentless 地主动"且"是一头野兽"。

Reddit 上一位有 9 年经验的老开发者 Bohdanowicz 的反馈尤为深刻。他此前花费数月打磨了一套多步骤工作流（任务分解、Harness 搭建等）来让旧 Claude 模型高效工作。Fable 5 发布后，这套精心构建的工作流变得基本不再必要——原本需要 4 到 6 个步骤和持续指导的任务，现在可以作为一个目标直接端到端执行。他的结论是："工作流的变化比模型本身的变化更大。但模型的进步是真实的。"

科技社区分析师 Matt Van Horn 分析了 865 条 Reddit 帖子，将社区共识总结为一个核心理念转变："直接把目标交给模型，不要指定怎么做。"Fable 5 的自主能力已经达到可以处理"做什么加完成标准"的级别，不再需要详细的逐步 SOP。

沃顿商学院教授 Ethan Mollick 用 Fable 5 从 2200 多个航班中提取数据并生成等时线地图，涵盖偏远地区的真实数据，还从单个初始提示生成了一款完整的可玩游戏。

Anthropic 官方编码页面引述了多个企业客户评价。Kay Zhu（联合创始人兼 CTO）称 Fable 5 在其内部评测中以一对一战胜所有被测模型排名第一，在 UI 设计和游戏编码这类最困难任务上领先尤为显著。一位 AI 平台客户报告代码审查反馈速度提升 60 倍，一家企业软件客户报告测试运行时间减少了 95%。

社区总结出的一条实用建议值得记录：不要用简单任务评测 Fable 5 与旧模型的差距，因为双方都能通过，学不到任何东西。应该找出当前模型持续失败的 3 个任务，各运行 5 次，对比完成率和每完成任务的成本。AtlasCloud 的专业建议是："将 Fable 5 用于你的 Agent 工作负载中最前沿最困难的 10% 到 20%，即 Opus 4.8 的失败率正在消耗你的重试和审查时间的那些任务；其余全部保持在 Opus 4.8 和 Sonnet 4.6。"

## 七、前端 vs 后端：场景差异

Fable 5 在不同编程领域的表现存在显著差异。

前端编码方面，Fable 5 表现尤为突出。在 Arena.ai 的 Code Arena 前端对决中，前端编程一对一对比胜率达 72%。开发者在 React、Vue 等现代前端框架的代码生成中普遍反馈，Fable 5 对组件生命周期、状态管理、性能优化的理解明显优于竞品。一位前端开发者评价其写的 React 代码不仅逻辑正确而且符合最佳实践，"像是有一个资深同事在旁边指导"。UI 设计能力也显著提升——在鱼皮的测试中，Fable 5 生成的看板页面状态区分清晰、颜色丰富、任务卡片信息排布合理，深色模式下的图表配色协调，整体效果是三个模型中最优的。

后端与系统编程方面，表现相对中规中矩。多位开发者反馈，在这些领域 Fable 5 与 Opus 4.8、GPT-5.5 的差距不大，更多是风格偏好而非能力差异。Fable 5 并非后端编程的绝对领先者，但因其长上下文保持能力，在处理跨服务调用、多系统集成等复杂后端架构任务时仍有优势。

文档与解释能力方面，提升明显。许多开发者提到 Fable 5 在生成代码文档和解释复杂逻辑方面有显著进步。它不仅告诉你代码做了什么，还会解释为什么这样做以及可能的替代方案。这对团队协作和知识传承非常有价值。

## 八、effort 参数：成本与能力的平衡术

Fable 5 的 effort 参数是实际使用中影响体验最直接的因素之一。

关键数据来自 SWE-bench Pro 评测：Fable 5 在 effort Low 档位下得分 75.0，而 Opus 4.8 在 xhigh 档位（最高努力程度）下得分仅为 68.6。换言之，Fable 5 的最低档表现已经超越了 Opus 4.8 的最高档。这颠覆了"花多少钱得到多少能力"的线性预期。

Claude Code 作者 Boris Cherny 对此给出了精确的技术解释：Fable 5 的单 token 定价确实是 Opus 4.8 的 2 倍，但完成同一任务平均消耗的 token 更少（官方数据为 25% 到 30% 的节省），综合下来实际成本反而可能更低。不过鱼皮的实测显示，在复杂长周期任务中，Fable 5 的实际花费仍是 Opus 4.8 的 3 倍（$38.66 vs $13.38），主要是因为 thinking token 消耗巨大。

effort Low 并非简单的"偷懒模式"，而是让模型聚焦核心推理路径、减少冗余计算。对于目标明确的任务如修复特定 Bug、重构特定模块，Low 档位已能覆盖需求；而对于需求分析、方案设计等模糊探索类任务，才需要更高的 effort 档位。这种按需分配算力的机制，从设计理念上更接近人类工程师的工作方式——不是每件事都用尽全力，而是把力气花在刀刃上。

## 九、安全护栏：编程可用性的最大障碍

Fable 5 在实际编程中最大的争议点不是能力不足，而是安全护栏对可用性的干扰。

最令人震惊的事件是 ProgramBench 评测。该基准由 SWE-Bench 作者团队出品，专注于从编译后的二进制文件重建源代码这一高难度任务。Fable 5 在 200 道题中全部拒绝作答，因为"二进制重建"操作触发了其网络安全分类器。尽管如此，ProgramBench 排行榜仍综合其他基准表现将 Fable 5 列在榜首——一个一字未答的考生坐了第一名的位置，这在 AI 评测史上绝无仅有。

Terminal-Bench 2.1 的数据更具体地揭示了护栏的影响范围：约 20.9% 的测试用例触发了安全拒绝并回退到 Opus 4.8。这意味着在实际编程任务中，每 5 个任务可能就有 1 个被降级处理。

在 Agents' Last Exam（ALE）评测中，Fable 5 得分 22.0% 排在 GPT-5.5 的 24.0% 之后，但每道题平均花费 15.70 美元，是 GPT-5.5 的 4 倍。在最高难度"Last-Exam"档位，包括 Fable 5 在内的所有模型通过率均为 0%。ALE 团队在排行榜备注中特别说明：Anthropic 可能默认悄无声息地提供模型的降级版本，因此评测数据可能低估了其真实能力。

反蒸馏机制的误触是另一个困扰。该机制旨在防止模型输出被用于训练竞争模型，但实际误触率远超官方宣称的 5%。开发者反馈中出现了仅询问普通编码问题就被路由回 Opus 4.8 的情况，甚至在简单打招呼时也触发保护机制。鱼皮测试时用 Fable 5 写文章也触发了安全过滤，被切换到 Opus 4.8。他在文章中吐槽："不是哥们，写个文章哪里不安全了？"

UC Berkeley RDI 实验室的 ALE 评测团队也在技术备注中警告："Fable 5 除了彻底的访问限制外，Anthropic 还可能默认悄无声息地提供该模型的一个降级版、低能力变体。重试无法纠正这一点，因此这里的数据可能低估了其真实能力——解读时需谨慎。"

AI 研究员 Nathan Lambert 的批评最为尖锐："一个在我不知情的情况下自动变笨的 AI 模型，从定义上就是不合规的 AI。"Fortune 杂志将 Fable 5 的安全降级机制描述为"秘密破坏"。压力之下，Anthropic 在发布后数小时内被迫修改政策——降级仍会发生，但会明确通知用户。

社区还观察到 Fable 5 的过度自主性问题。Simon Willison 花了 12 美元修复一个 CSS 滚动条 Bug，因为模型没有简单地修补一行代码，而是自主搭建了一整套跨浏览器测试环境来验证修复效果。Bill and Melinda Gates 基金会的一位流行病学家报告，安全分类器在其首日使用的几乎每个会话中都触发，Anthropic 后来将触发率调整至约 0.05%。

## 十、社区共识：适用场景总结

综合开发者社区的实测反馈，以下是 Fable 5 在实际编程中的适用场景画像。

在以下场景中，Fable 5 展现出了明显优势：大规模代码迁移与重构，尤其是跨语言迁移（如从 AngularJS 迁移到 React）或者跨版本升级；遗留系统治理，需要对存量代码做全局理解和依赖关系分析；长周期自主开发，可以写完整任务简报后离开数小时等待结果；多文件跨模块功能开发，需要保持跨文件一致性和相互引用正确性；安全漏洞修复，特别是在需要同时理解攻击原理和修复策略的复杂漏洞场景；前端现代框架开发，尤其是 React 和 Vue 的组件级代码生成。

在以下场景中不建议使用 Fable 5 或其优势不明显：日常 Bug 修复和单文件编辑，Opus 4.8 已经足够优秀且性价比更高；交互式探索性编程，需要快速迭代反馈；涉及二进制逆向、网络安全评估等可能触发安全分类器的任务，直接使用 Opus 4.8 可避免降级和往返延迟；对成本极其敏感的场景，单次复杂任务可能花费 15 美元以上；后端服务和系统编程的常规开发，与 Opus 4.8 差距不大。

中文技术社区的形成共识是：默认使用 Opus 4.8，将最困难的 10% 到 20% 任务升级至 Fable 5。对 2026 年 80% 的团队来说，Opus 4.8 就是正确的默认答案。

## 十一、结论

Claude Fable 5 是当前编程能力最强的模型这一命题需要加一个限定词：在长周期、高复杂度的编程任务中。它在短平快的日常编码场景中并未带来革命性的体验提升，但在迁移重构、遗留代码治理和自主长周期开发等任务上创造了质的飞跃。

鱼皮的首轮全栈测试中，Fable 5 是唯一零修改跑通的模型，交付确定性远超竞品。次轮 Claude Code 源码重构测试中，它是唯一交付可用产品的模型，且在真实终端环境中做了 PTY 交互调试。Endor Labs 安全修复评测中，它解出 4 道无模型能解的漏洞题。但所有这些优势都伴随着显著成本——实际任务的费用是 Opus 4.8 的 3 到 8 倍。

Fable 5 最大的问题不是能力，而是可用性。安全护栏过于粗糙导致 ProgramBench 200 题全部拒答、Terminal-Bench 20.9% 触发降级、反蒸馏机制频繁误触、以及曾偷偷降级不通知用户的设计，这些都严重侵蚀了用户对模型的信任。一个什么都懂但很多都不说的超能助手，在编程场景中产生的挫败感可能比一个平庸但透明的助手更强。

Fable 5 与其说是编程工具的终极形态，不如说是展示了下一代 AI 编程助手的可能方向。它证明了 AI 不只可以帮你写代码，更可以帮你解决问题——正如 Bohdanowicz 所感悟的，过去需要数月打磨的多步骤工作流，在 Fable 5 面前变得不再必要。但当安全护栏的边界找准了、当成本进一步下探了、当它与 Opus 4.8 的分工更清晰了——那时才是 Fable 和 Opus 双线策略真正释放价值的时刻。

Fable 5 被美国政府出口管制指令强制停用外国用户访问（2026 年 6 月 11 日）的影响已超出技术范畴。仅 4 天的可用窗口使大量开发者仅能进行初步体验，无法形成系统性评估。中国企业智谱 AI 于 6 月 15 日发布 GLM-5.2，明确指出 Fable 5 的停用证明了"美国 AI 模型不能作为国际客户的可依赖选项"——这标志着 AI 编程工具的地缘政治竞争已正式开启。
